## Why no migrations were shipped in this audit

- **Zero parent USs in `done` state** — there is no US where the only blocker
  is "schema migration needs writing." Shipping schema ahead of the feature
  consuming it would create dead tables.
- **Estimated scope** — at an avg ~3.5h per migration (range 2h–4h per
  scaffold estimate), the 63 real-work items represent ~220 engineering
  hours. That is a multi-sprint program, not a single audit pass.
- **Migration tooling is in place** — Alembic is configured at
  `/alembic.ini` with versions under `alembic/versions/`. When the
  parent USs are picked up, scaffolded migrations should be authored there,
  not as one-shot SQL.

## Remaining open count

After this audit:
- Schema-migration todos in `todo` status: **63** (down from 65)
- All 63 are correctly bound to unshipped parent USs

## Recommendation

When a parent US is moved to `in_progress`, its associated schema-migration
child should move with it. Group migrations by parent US into a single
Alembic revision file under `alembic/versions/` (do not create one revision
per child task — that fragments the schema timeline).
